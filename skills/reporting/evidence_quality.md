# Objective

## Changelog

### 1.0.0
- Initial version created for the offensive operator skill library.

Assess the quality of collected evidence across all executed phases (P01–P22) and generate the final offensive campaign narrative. The report covers: phases executed, skills used per phase, MCP tool executions, evidence strength per finding, hypotheses lifecycle, validated attack chains, and executive risk summary. This skill does NOT generate imaginary findings — it compiles and validates only what was observed.

# When To Use

Select this skill when:
- `pentest_phase_index` is at P22 (final reporting phase)
- All active phases have status `completed|partial|failed|skipped` in `phase_ledger`
- `vulnerabilidades_encontradas` has been populated by vulnerability phases
- Phase P22 is the current `current_pentest_phase_id`

# Preconditions

- `phase_ledger` contains records for all attempted phases
- `attack_paths` contains any confirmed attack chains
- `active_hypotheses` have been resolved (validated/discarded/inconclusive)
- Executive summary template available

# Evidence Quality Model

Evidence strength levels used throughout the report:

| Level | Definition | Example |
|-------|-----------|---------|
| `conclusive` | Independently reproducible, multiple confirmations | SQLi with DB dump proof + screenshot |
| `strong` | Single reproduction with tool output artifact | XSS with alert() proof-of-concept URL |
| `medium` | Tool output without manual verification | Port scan result with banner |
| `weak` | Inference or circumstantial signal | Server header reveals version |
| `none` | Tool ran but no finding produced | Phase completed, 0 vulnerabilities |

Minimum evidence for reporting:
- Critical/High findings: `strong` or `conclusive`
- Medium findings: `medium` or above
- Low findings: `weak` or above
- Informational: any evidence type accepted

# Report Structure

The report generated by this skill follows this structure:

## Section 1: Campaign Overview
- Target, authorization scope, scan mode
- Campaign stage reached
- Noise profile used
- Total phases executed vs. planned
- Execution timeline (start → end)

## Section 2: Phase Execution Journal (P01–P22)
For each phase:
- Phase ID + name
- Status: completed/partial/failed/skipped
- Skills invoked
- Tools executed via MCP
- Evidence collected (count + strength)
- Time taken
- Exit criteria met/not met
- Skip reason (if skipped)

## Section 3: Asset Inventory
- Subdomains discovered (P01)
- Live hosts and open ports (P02)
- HTTP/HTTPS endpoints (P02–P03)
- Technologies fingerprinted
- Admin panels found

## Section 4: Vulnerability Findings
For each finding:
- Vulnerability class + CVE if applicable
- Severity (Critical/High/Medium/Low/Informational)
- Phase where found
- Skill that discovered it
- Tool used + MCP execution ID
- Evidence strength
- Affected endpoint
- Reproduction steps
- Recommended remediation

## Section 5: Attack Chain Analysis
For each validated chain:
- Chain name + description
- Phases involved
- Confidence score
- CVSS estimate
- Step-by-step path
- Evidence IDs supporting each step
- Business impact

## Section 6: Hypothesis Lifecycle
For each hypothesis:
- Hypothesis statement
- Created by (skill/signal)
- Test phases
- Final status: validated/discarded/inconclusive
- Evidence that resolved it

## Section 7: Risk Rating
- Overall EASM risk score
- Distribution by severity
- Most critical asset
- Most critical attack path
- Comparison with industry baseline (if available)

## Section 8: Recommendations
- Critical (fix immediately): list
- High (fix within 30 days): list
- Medium (fix within 90 days): list
- Structural recommendations (architecture/process)

# Evidence Validation Rules

Before including a finding in the report:

1. **Reproducibility check**: Was the finding confirmed by at least 1 tool execution? (evidence_strength ≥ weak)
2. **False positive control**: Was a negative control attempted? (e.g., different parameter, baseline request)
3. **Phase traceability**: Is there a `phase_ledger` entry with `evidence_ids` linking to this finding?
4. **Skill attribution**: Is there a `skill_id` associated with the finding?
5. **MCP traceability**: Is there an `mcp_request_id` for the tool execution that found it?

Findings that fail these checks are marked `unverified` and listed separately with a note.

# Evidence Strength Scoring

Calculate per-finding evidence strength:

```python
def score_evidence_strength(finding: dict) -> str:
    score = 0
    if finding.get("tool_output_artifact"):     score += 2
    if finding.get("oob_callback_confirmed"):   score += 3
    if finding.get("manual_verification"):      score += 2
    if finding.get("reproduction_count", 0) > 1: score += 1
    if finding.get("negative_control"):         score += 1
    
    if score >= 6: return "conclusive"
    if score >= 4: return "strong"
    if score >= 2: return "medium"
    if score >= 1: return "weak"
    return "none"
```

# Validation Logic

**SUCCESS**: Report generated with at least 1 finding documented, phase ledger covers all attempted phases

**PARTIAL**: Report generated but some phases have no evidence — documented as `skipped` with reason

**FAILURE**: phase_ledger is empty or corrupt, no findings exist after all phases

# Quality Gate Rules

The following findings are BLOCKED from appearing in the final report:

1. Findings with `evidence_strength: none` — cannot report unconfirmed findings
2. Findings without a `phase_id` — must be traceable to a phase
3. Findings without a `skill_id` — must be attributed to a skill
4. Findings generated by LLM inference alone (no tool execution) — must have tool artifact
